# ADR: 토큰 저장 전략 — sessionStorage vs HttpOnly Cookie

**날짜:** 2026-06-23
**상태:** 채택됨

## 결정
관리자 JWT는 sessionStorage에 저장하되, 타액티브 타임아웃(20분)과 주기적 갱신(1분 스로틀)으로 보안을 강화한다.

## 배경
관리자 대시보드는 고도의 민감한 데이터(지원자 정보, 시스템 통계)에 접근하므로 인증 토큰의 보안이 매우 중요하다. 토큰 저장소 선택은 XSS 취약성, 탭 격리, 세션 생명 주기 등 여러 요소를 고려해야 한다.

## 비교

| 항목 | sessionStorage (현재) | HttpOnly Secure Cookie |
|------|----------------------|------------------------|
| XSS 토큰 탈취 | 가능 (JS 접근 가능) | 불가능 (JS 차단) |
| 탭 간 격리 | 각 탭 독립적 | 도메인 공유 |
| 브라우저 닫기 시 | 자동 삭제 | 설정에 따름 |
| CSRF 위험 | 매우 낮음 | 중간 (추가 방어 필요) |
| 구현 복잡도 | 낮음 | 높음 (백엔드 조정 필요) |
| Supabase 지원 | 네이티브 지원 | 제한적 |

## 결정 이유

1. **XSS 취약성의 기존 방지 전략** — 현재 Vite 빌드는 기본적으로 XSS 방어를 포함하고, 사용자 입력은 React의 기본 이스케이핑 메커니즘으로 보호된다. 서드파티 의존성을 최소화하여 XSS 공격 면을 줄였다.

2. **타액티브 타임아웃** — 20분 자동 로그아웃은 토큰 탈취 후 악용 시간 윈도우를 극도로 제한한다. 이는 HttpOnly Cookie의 보안 이점 상당 부분을 상쇄한다.

3. **탭 격리 및 세션 정책** — 관리자는 단일 활성 세션만 필요하므로 sessionStorage의 탭 간 격리가 추가 이점이다. 탭 하나를 닫으면 해당 탭의 토큰이 즉시 사라진다.

4. **백엔드 통합 단순성** — HttpOnly Cookie 전환 시 Supabase Edge Function의 쿠키 처리 로직, 도메인/경로 설정, CSRF 토큰 관리 등이 필요하다. 현재 Bearer 토큰 방식은 이미 검증되고 운영 중이다.

## 현재 완화책

- **타액티브 타임아웃** — 세션 비활성 시 자동 로그아웃 (20분)
- **주기적 갱신** — 사용자 활동 감지 시 토큰 자동 갱신 (1분 스로틀로 과부하 방지)
- **활동 이벤트 모니터링** — mousemove, keydown, click, scroll, touchstart로 사용자 상태 추적
- **로그아웃 시 백엔드 알림** — /admin-logout 엔드포인트 호출로 서버 세션 무효화
- **Bearer 토큰 방식** — Authorization 헤더를 통한 명시적 인증으로 자동 전송 불가

## 향후 검토 조건

- **XSS 사건 발생** — 성공한 XSS 공격이 발생하면 HttpOnly Cookie로의 전환을 우선 검토
- **다중 탭 지원 요구** — 관리자가 여러 탭에서 동시 세션을 요청하면 쿠키 기반으로의 변경 고려
- **Supabase 공식 HttpOnly 지원** — Supabase가 HttpOnly 쿠키 관리 라이브러리를 제공하면 마이그레이션 비용 재평가
